## Issue 9: MLflow container stuck on `pip install` at startup
**Problem:** MLflow container showed `Up (health: starting)` but never became healthy. `docker logs` showed no output. `curl http://localhost:5000/health` returned empty replies.

**Root Cause:** The `docker-compose.yml` used `image: python:3.11-slim` with `command: bash -c "pip install ... && mlflow server ..."`. Pip output is buffered in non-interactive mode, so no logs appeared. More critically, the install runs **every time the container starts**, making it slow (~80s) and fragile.

**Previous setup that "worked":** In the 2026-05-05 e2e test, MLflow was running from a previous restart where pip install had already completed. On a clean restart, it always hangs during pip install.

**Why local client testing failed:** Running `python ml/train_model.py` from the host with `MLFLOW_TRACKING_URI=http://localhost:5000` caused `PermissionError: [Errno 13] Permission denied: '/opt/ml'`. With `file://` artifact storage, the client must have access to the same filesystem path ‚Äî `/opt/ml/artifacts` only exists inside containers.

**Fix (3 parts):**
1. **Created `Dockerfile.mlflow`** ‚Äî Pre-bakes ML packages at build time, not runtime:
   ```dockerfile
   FROM python:3.11-slim
   RUN pip install --no-cache-dir mlflow==2.15.0 scikit-learn xgboost pandas psycopg2-binary setuptools
   CMD ["mlflow", "server", "--host", "0.0.0.0", "--port", "5000", \
        "--backend-store-uri", "sqlite:////opt/scripts/mlflow.db", \
        "--default-artifact-root", "file:///opt/ml/artifacts"]
   ```

2. **Updated `docker-compose.yml`** ‚Äî Changed MLflow service from `image: python:3.11-slim` to `build: Dockerfile.mlflow`. Removed inline pip install command.

3. **Added `./ml:/opt/ml` volume mount to Airflow containers** ‚Äî Both webserver and scheduler now have the MLflow artifact path mapped, so MLflow client inside Airflow can write artifacts to the same filesystem as the MLflow server.

**Key Takeaway:** Never run `pip install` in container `command:` or `entrypoint:`. Always bake dependencies at build time. With `file://` artifact storage, all clients and the server must share the same filesystem path.

## Issue 10: Metabase MBQL `filters` vs `filter` key mismatch
**Problem:** Cards created via API returned 400 errors with SQL generation failures: `ERROR: argument of AND must be type boolean, not type character varying`.

**Root Cause:** The MBQL query builder used `filters` (plural) as the key in the stage object:
```python
# WRONG ‚Äî Metabase silently accepts this but generates broken SQL
stage["filters"] = ["=", ["field", ...], "hackernews"]
```

**Correct key is `filter` (singular):**
```python
# CORRECT ‚Äî Metabase generates valid SQL
stage["filter"] = ["=", ["field", ...], "hackernews"]
```

**Impact:** All cards with WHERE clauses (HN filter, YT filter, GH filter, combined filters) produced invalid SQL. Cards without filters worked fine.

**Fix:** Changed `stage["filters"]` to `stage["filter"]` in `make_mbql()` function.

## Issue 11: MBQL field IDs stale after table sync ‚Äî 403 permission errors
**Problem:** After fixing `filter` key, several cards failed with 403 errors: `You cannot save this Question because you do not have permissions to run its query`. Affected tables: `gold_trending_topics`, `dim_creators`, `fct_daily_engagement`.

**Root Cause:** MBQL queries reference columns by numeric field IDs (e.g., `115` for `total_items`). These IDs are assigned during Metabase's table sync and can change between syncs. Hardcoded IDs in the script became stale, causing Metabase to reject the queries as invalid (the field ID doesn't map to a valid column in the synced schema).

**Investigation:**
- Tables that worked: `gold_individual_items` (id=9) ‚Äî synced first, IDs stable
- Tables that failed: `gold_trending_topics` (id=16), `dim_creators` (id=13) ‚Äî synced later, different ID ranges
- Even correct IDs (verified via `/table/{id}/query_metadata`) still failed ‚Äî Metabase's permission system rejects queries when field IDs don't match the internal schema cache

**Fix:** Converted ALL cards from MBQL to native SQL queries. SQL queries reference columns by name, not by numeric ID, so they never go stale:
```python
# Before (fragile ‚Äî depends on synced field IDs):
make_mbql(db_id, table_id,
    aggregation=[["sum", ["field", 143, {"base-type": "type/BigInteger"}]]],
    breakout=[["field", 142, {"base-type": "type/Text"}]])

# After (reliable ‚Äî uses column names directly):
create_sql_card(session, "Top 20 Words", "bar", db_id,
    "SELECT word, SUM(frequency) FROM gold_trending_topics GROUP BY word ORDER BY SUM(frequency) DESC LIMIT 20")
```

**Key Takeaway:** When creating Metabase cards programmatically via API, **always use native SQL queries** (`"type": "native"`) instead of MBQL. MBQL field IDs are fragile and break after any table schema change or re-sync. SQL is stable, readable, and works regardless of sync state.
