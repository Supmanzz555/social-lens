## Issues Encountered & Fixes

### Issue 1: `kafka-python` missing from Airflow container
**Problem:** Producers failed inside Airflow container with `ModuleNotFoundError: No module named 'kafka'`. The `Dockerfile.airflow` installs `pyspark boto3 requests asyncpg great-expectations` but NOT `kafka-python`.
**Root Cause:** `kafka-python` was likely manually pip-installed in a previous session but not persisted in the Dockerfile. When containers were rebuilt, the package was lost.
**Fix:** Added `kafka-python` to `Dockerfile.airflow` line 7:
```dockerfile
RUN pip install --no-cache-dir pyspark==3.5.1 boto3 requests asyncpg great-expectations kafka-python \
```
Rebuilt Airflow image: `docker compose build airflow-webserver airflow-scheduler`
**Key Takeaway:** All Python dependencies used by DAG tasks must be in the Dockerfile. Manual `pip install` in running containers is lost on rebuild.

### Issue 2: `scripts/` directory not mounted to Airflow containers
**Problem:** `validate_silver_quality` task failed with `can't open file '/opt/airflow/scripts/run_data_quality.py': [Errno 2] No such file or directory`.
**Root Cause:** The `batch_pipeline.py` DAG references `cd /opt/airflow && python scripts/run_data_quality.py`, but the `scripts/` host directory was not mounted into the Airflow containers. Only `dags/`, `plugins/`, `logs/`, `spark/`, `configs/`, and `producers/` were mounted.
**Fix:** Added `- ./scripts:/opt/airflow/scripts` to both `airflow-webserver` and `airflow-scheduler` volumes in `docker-compose.yml`.
**Key Takeaway:** Any host directory referenced by DAG tasks must be explicitly mounted in docker-compose.

### Issue 3: `data_quality/` directory not mounted to Airflow containers
**Problem:** After fixing Issue 2, `validate_silver_quality` still failed with `FileNotFoundError: Expectations file not found: /opt/airflow/data_quality/expectations/youtube_expectations.json`.
**Root Cause:** `run_data_quality.py` reads expectation files from `/opt/airflow/data_quality/expectations/`, but the `data_quality/` directory was not mounted.
**Fix:** Added `- ./data_quality:/opt/airflow/data_quality` to both Airflow service volumes in `docker-compose.yml`.
**Key Takeaway:** Same as Issue 2 — verify all file paths used by DAG tasks have corresponding volume mounts.

### Issue 4: Scheduled DAG runs blocking manual triggers
**Problem:** After restarting containers, `batch_pipeline` had a `scheduled__2026-05-04T02:00:00+00:00` run that was queued/running. Manual trigger (`max_active_runs=1`) got stuck behind it, appearing to hang in "queued" state indefinitely.
**Root Cause:** `start_date=datetime(2026, 1, 1)` + `catchup=False` still creates one scheduled run for the most recent interval. With `max_active_runs=1`, the scheduled run blocks manual triggers.
**Fix:** Before triggering manually, delete scheduled runs:
```sql
DELETE FROM task_instance WHERE run_id LIKE 'scheduled%';
DELETE FROM dag_run WHERE run_id LIKE 'scheduled%';
UPDATE dag SET is_paused=true WHERE dag_id IN ('ingest_streaming', 'batch_pipeline');
```
Then unpause and trigger. Or set `catchup=False` and ensure DAGs are paused on creation (`AIRFLOW__CORE__DAGS_ARE_PAUSED_AT_CREATION: 'true'`).
**Key Takeaway:** Always clean scheduled runs before manual triggers on `max_active_runs=1` DAGs.

## Best Practices for Clean Restart (2026-05-05)

### Changes Made to Prevent Future Errors

#### 1. Metabase on data-platform network (docker-compose.yml)
**Problem:** After container restart, Metabase couldn't resolve `postgres` hostname → `UnknownHostException` when adding database.
**Fix:** Added `networks: [data-platform]` and `depends_on: postgres` to metabase service. No more manual `docker network connect` needed.

#### 2. Metabase dashboard creation via POST with inline cards (setup_metabase.py)
**Problem:** `PUT /dashboard/{id}` with negative card IDs caused H2 auto-increment sequence collision → "Primary key violation" errors.
**Fix:** Create cards first via `POST /card`, then create dashboard with cards inline via single `POST /dashboard` with `dashcards` array. H2 allocates IDs correctly on INSERT.

#### 3. DAGs start paused (docker-compose.yml)
**Problem:** `start_date=datetime(2026, 1, 1)` + `catchup=False` still creates one scheduled run on first load. With `max_active_runs=1`, it blocks manual triggers.
**Fix:** Changed `AIRFLOW__CORE__DAGS_ARE_PAUSED_AT_CREATION: 'true'`. DAGs load paused, no scheduled runs. Unpause manually in UI when ready to test.

#### 4. All dependencies baked into Dockerfile.airflow
**Problem:** Manual `pip install` is lost on image rebuild. `kafka-python` and ML packages were missing after rebuild.
**Fix:** All packages in single RUN layer:
```dockerfile
RUN pip install --no-cache-dir pyspark==3.5.1 boto3 requests asyncpg great-expectations kafka-python \
    && pip install --no-cache-dir apache-airflow-providers-apache-spark==4.1.5 --no-deps \
    && pip install --no-cache-dir apache-airflow-providers-openlineage==1.8.0 --no-deps \
    && pip install --no-cache-dir mlflow>=2.11.0 scikit-learn>=1.4.0 xgboost>=2.0.3 pandas>=2.2.0 psycopg2-binary
```

#### 5. ML directory mounted to Airflow containers
**Problem:** `ml_pipeline` DAG references `/opt/airflow/ml/` but directory wasn't mounted.
**Fix:** Added `- ./ml:/opt/airflow/ml` to both webserver and scheduler volumes.

#### 6. Idempotent setup_metabase.py
- Checks for existing database before creating
- Creates all 5 dashboards in one pass (including ML if table exists)
- Gracefully skips ML dashboard if `ml_predictions` table doesn't exist yet
- Handles both `{"data": [...]}` and raw list API responses

## ML Pipeline Issues (2026-05-05)

### Issue 5: `source .env` fails inside Airflow container
**Problem:** `ml_pipeline` DAG tasks use `source .env && python ml/...` but `.env` doesn't exist inside the container → exit code 1.
**Root Cause:** `.env` is a host file loaded by docker-compose as environment variables. It's not a file inside the container.
**Fix:** Remove `source .env` from `ml_pipeline.py` bash commands. Environment variables are already injected by docker-compose `env_file`.

### Issue 6: `ml/data/` permission denied (UID 50000)
**Problem:** `generate_synthetic_data` task fails with `PermissionError: [Errno 13] Permission denied: 'ml/data/synthetic.csv'`.
**Root Cause:** Airflow runs as UID 50000. Host `ml/data/` directory is owned by host user (UID 1000). Bind mount inherits host ownership.
**Fix:** `chmod -R 777 ml/` on host.
**Key Takeaway:** Any mounted volume that Airflow tasks write to must be world-writable or match UID 50000.

### Issue 7: MLflow connection refused (localhost:5000 from container)
**Problem:** `train_model` task fails with `ConnectionError: HTTPConnectionPool(host='localhost', port=5000): Connection refused`.
**Root Cause:** `MLFLOW_TRACKING_URI=http://localhost:5000` in `.env`. Inside the Airflow container, `localhost` refers to the container itself, not the host machine where MLflow runs.
**Fix Attempts (all failed):**
1. `MLFLOW_TRACKING_URI=http://host.docker.internal:5000` — MLflow rejects with "Invalid Host header - possible DNS rebinding attack detected"
2. `host.docker.internal:host-gateway` via `extra_hosts` — MLflow DNS rebinding protection still blocks the request
3. Running MLflow on host with `--expose-pypi-packages` flag — flag doesn't exist in MLflow 3.x
**Root Problem:** MLflow 3.11.1 has strict DNS rebinding protection that rejects `host.docker.internal` headers. No flag to disable it in current version.
**Pending Fix:** Run MLflow as a Docker container on the same `data-platform` network so Airflow can reach it via `http://mlflow:5000` (no DNS rebinding issue).

### Issue 8: MLflow server unstable on host
**Problem:** MLflow 3.11.1 spawns multiple worker processes that crash/shutdown randomly. `pkill -f mlflow` hangs indefinitely.
**Root Cause:** MLflow 3.x uses gunicorn/uvicorn with multiple worker processes. Signal handling is broken on some Linux kernels.
**Impact:** Cannot reliably start MLflow on host for Airflow container access.
