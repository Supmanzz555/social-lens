## Issues Encountered & Fixes

### Issue 1: Producers couldn't reach Kafka from Airflow container
**Problem:** `docker exec airflow-webserver python producers/hackernews_producer.py` failed with `NoBrokersAvailable`.
**Root Cause:** Inside the container, `localhost:9092` refers to the Airflow container itself, not the Kafka container. The `.env` variable `KAFKA_BOOTSTRAP_SERVERS_LOCAL=localhost:9092` is configured for the **host machine** context.
**Fix:** Run producers from the host machine using `.venv`. Docker port mapping (`9092:9092`) makes `localhost:9092` on the host reach Kafka.
**Rule Added:** AGENT.md Rule 15 — "Always follow MASTER_PLAN.md verification steps exactly. Do not run host scripts inside Docker containers."

### Issue 2: gold_to_postgres failed with Connection Refused
**Problem:** Task showed `success` but Postgres had 0 rows. Logs showed `java.net.ConnectException: Connection refused` to `localhost:5432`.
**Root Cause:** `gold_to_postgres.py` hardcoded `PG_HOST = "localhost"`. Inside Airflow container, `localhost` is the Airflow container, not the Postgres container.
**Fix:** Changed to `PG_HOST = os.getenv("PG_HOST", "localhost")` and passed `env_vars={"PG_HOST": "postgres"}` in the DAG's `SparkSubmitOperator`.
**Additional Fix:** Added error handling so the script exits with code 1 on failure instead of silently succeeding.

### Issue 3: Duplicate data on re-run
**Problem:** Running `batch_pipeline` twice doubled the Postgres row counts.
**Root Cause:** Spark JDBC used `mode("append")` which always adds rows.
**Fix:** Added `psycopg2` connection to `gold_to_postgres.py` that does `DELETE FROM <table>` before Spark writes new data.

### Issue 4: Kafka custom command bypassed env vars
**Problem:** `KAFKA_ADVERTISED_LISTENERS` env var was set but not applied because the custom `command:` in docker-compose.yml bypassed the Kafka entrypoint script.
**Fix:** Removed the custom command from Kafka service. The default entrypoint handles KRaft formatting and topic creation automatically.
**Revert:** Later reverted docker-compose.yml to original because the issue was actually just running producers from the wrong place.

## Issues Encountered & Fixes (continued)

### Issue 5: Streaming consumed messages but didn't write them to bronze
**Problem:** Kafka had 210 HN, 187 GH, 199 YT messages but `bronze/` only had files with ~10 items each after streaming.
**Root Cause:** Spark Streaming checkpoints track offsets. When the first streaming run stopped after 45s, it had consumed all messages but checkpointed them before writing the final batch to R2. Subsequent runs saw the checkpoint ("already consumed") and read nothing.
**Fix:** Delete `spark/streaming/checkpoints/` directory AND existing `bronze/` files before re-running streaming. This forces Spark to re-read from `earliest` offsets and write everything.

### Issue 6: GE validation failed on GitHub `expect_column_values_to_be_in_set`
**Problem:** `validate_silver_quality` task failed on GitHub data.
**Root Cause:** The expectations file only listed 7 event types (`PushEvent`, `WatchEvent`, `ForkEvent`, `PullRequestEvent`, `IssuesEvent`, `CreateEvent`, `DeleteEvent`), but real GitHub data included 3 additional types: `IssueCommentEvent`, `PullRequestReviewEvent`, `ReleaseEvent`.
**Fix:** Updated `data_quality/expectations/github_expectations.json` to include all 10 event types. **Lesson:** GE expectations must cover all real-world values, not just expected ones.

### Issue 7: gold_to_postgres failed with "cannot drop table because other objects depend on it"
**Problem:** After switching from `mode("append")` to `mode("overwrite")`, the task failed on `gold_content_engagement`.
**Root Cause:** Spark JDBC `overwrite` mode does `DROP TABLE` + `CREATE TABLE` internally. The `gold_content_engagement` table is referenced by dbt models (views), so Postgres refuses to drop it.
**Fix:** Reverted to `mode("append")` with `DELETE FROM` via psycopg2 before loading. This preserves the table structure that dbt views depend on.

### Issue 8: Metabase dashboard cards return "API endpoint does not exist"
**Problem:** `POST /api/dashboard/{id}/cards` returned 404.
**Root Cause:** Metabase v0.60+ removed the dedicated add-cards endpoint. Cards must be added by updating the entire dashboard via `PUT /api/dashboard/{id}` with a full `dashcards` array. Each card needs a negative `id` (e.g., `-1`, `-2`) to signal it's a new entry.
**Fix:** Use `PUT /api/dashboard/{id}` with complete dashcards list.

### Issue 9: Metabase container can't reach Postgres
**Problem:** API returned "The connection attempt failed" when adding Postgres via Metabase UI.
**Root Cause:** Metabase container was on the `default` network while Postgres was on `data-platform`. Containers on different Docker networks can't resolve each other by name.
**Fix:** `docker network connect social-media-data-platform_data-platform metabase`

### Issue 10: dotenv not loading on host
**Problem:** Running Python scripts on host with `os.getenv()` returned `None` for R2 credentials.
**Root Cause:** `os.getenv()` reads from shell environment. `.env` file is not automatically loaded. Must use `from dotenv import load_dotenv; load_dotenv()` before calling `os.getenv()`.
**Fix:** Always use `dotenv` for local scripts. Inside Airflow containers, env vars are injected by docker-compose.

### Issue: _spark_metadata breaks bronze_to_silver
**Problem**: `batch_pipeline` first attempt failed on `validate_silver_quality` (exit code 1). Root cause was upstream: `bronze_to_silver` reported success but wrote 0 silver rows. Spark error: `IllegalStateException: s3a://datalake/bronze/source=hackernews/_spark_metadata/0 doesn't exist`.
**Root Cause**: Spark Streaming writes `_spark_metadata/` directory with compacted file logs (entry 1, 2, ...). When Spark batch job runs `spark.read.parquet()` on the same path, it detects `_spark_metadata/` and switches to the streaming file index, which expects entry 0. Since we cleaned all bronze files and re-streamed, only entry 1 existed — no entry 0.
**Fix**: Delete `_spark_metadata/` directories from R2 bronze before running batch pipeline:
```python
for source in ['hackernews', 'github', 'youtube']:
    prefix = f'bronze/source={source}/_spark_metadata/'
    resp = s3.list_objects_v2(Bucket=bucket, Prefix=prefix)
    for o in resp.get('Contents', []):
        s3.delete_object(Bucket=bucket, Key=o['Key'])
```
**Key Takeaway**: Spark batch and streaming jobs share the same bronze path. Streaming leaves metadata that breaks batch reads. Clean `_spark_metadata/` as part of the reset procedure, or have `bronze_to_silver.py` read individual files explicitly instead of scanning the directory.

## Issues Encountered: ingest_streaming DAG Fix (2026-05-03)

### Issue 12: Kafka advertised.listeners returns localhost:9092 to clients inside container
**Problem:** Producer connects to `kafka:9092` successfully, but Kafka's broker metadata advertises `localhost:9092`, causing the client to reconnect to `localhost` which fails inside the container.
**Root Cause:** Kafka's custom `command:` in docker-compose.yml bypasses the entrypoint script that processes `KAFKA_ADVERTISED_LISTENERS` env var. The config file (`/opt/kafka/config/kraft/server.properties`) has `advertised.listeners=PLAINTEXT://localhost:9092` hardcoded.
**Fix:** 
1. Added `sed -i` command to docker-compose.yml Kafka `command:` to rewrite advertised.listeners before startup
2. For running container: `docker exec kafka sed -i 's|advertised.listeners=PLAINTEXT://localhost:9092|advertised.listeners=PLAINTEXT://kafka:9092|' /opt/kafka/config/kraft/server.properties`
3. Restarted Kafka with fresh volume (cluster ID mismatch after manual edit)
**Key Takeaway:** Custom commands bypass env var processing. If you override the entrypoint, you must manually handle config file edits.

### Issue 13: Permission denied on producers/.hn_last_id and spark checkpoints
**Problem:** `PermissionError: [Errno 13] Permission denied: '/opt/airflow/producers/.hn_last_id'` and similar for Spark checkpoints.
**Root Cause:** Mounted volumes from host are owned by host user, Airflow container runs as `airflow` (UID 50000). Files created by previous runs (or host user) aren't writable.
**Fix:** `chmod -R 777` on `producers/` and `spark/streaming/checkpoints/` on host.
**Key Takeaway:** Any mounted volume that Airflow tasks write to must be world-writable or match UID 50000.

### Issue 14: Spark streaming CONCURRENT_STREAM_LOG_UPDATE
**Problem:** `Multiple streaming jobs detected for 0` / `Concurrent update to the log` — Spark refuses to start because previous streaming run's checkpoints indicate an active stream.
**Root Cause:** Failed Spark processes from previous DAG runs remain alive inside the container, holding checkpoint locks. Also, stale checkpoint files from previous runs cause conflicts.
**Fix:** Kill stale Spark processes: `docker exec airflow-webserver bash -c "kill -9 \$(pgrep -f 'spark-submit') \$(pgrep -f 'org.apache.spark')"`. Clean checkpoints: `rm -rf /opt/airflow/spark/streaming/checkpoints/*`.
**Key Takeaway:** Spark streaming checkpoints track active stream state. Always clean checkpoints AND kill processes before retrying.

### Issue 15: `_spark_metadata` in bronze breaks batch reads
**Problem:** Spark Streaming writes `_spark_metadata/` directory to `bronze/source=X/`. When `bronze_to_silver.py` runs `spark.read.parquet()` on the same path, Spark detects the metadata dir and switches to the streaming file index, which expects entry 0 (missing after clean slate).
**Root Cause:** Batch and streaming jobs share the same bronze path. Streaming leaves metadata that breaks batch reads.
**Fix (proper):** Staging → commit pattern. Streaming writes to `bronze/staging/source=X/`. A `commit_bronze` task copies parquet files from staging to `bronze/source=X/`. Batch reads clean `bronze/source=X/` (no `_spark_metadata`).
**Implementation:**
- `kafka_to_r2.py`: Changed `r2_path` from `bronze/source={source_name}` to `bronze/staging/source={source_name}`
- `ingest_streaming.py`: Added `commit_bronze` task that `copy_object` + `delete_object` per parquet file
- Pipeline: `[producers] → spark_kafka_to_r2 → commit_bronze`
- `batch_pipeline.py`: Removed the `clean_spark_metadata` workaround hack
**Key Takeaway:** Separate streaming write path from batch read path. This is the production pattern — staging → commit, not delete-metadata.

### Issue 16: Concurrent DAG runs cause Spark conflicts
**Problem:** `max_active_runs` defaults to 16. If multiple `ingest_streaming` runs overlap, Spark streaming jobs write to the same checkpoints → `CONCURRENT_STREAM_LOG_UPDATE`.
**Root Cause:** Default Airflow DAG config allows parallel runs. Spark streaming doesn't support concurrent writes to same checkpoint dir.
**Fix:** Set `max_active_runs=1` on both `ingest_streaming` and `batch_pipeline` DAGs. Next run queues until current finishes.
**Key Takeaway:** DAGs with shared state (checkpoints, same write paths) must be serialized. `max_active_runs=1` is the standard pattern for streaming DAGs.

### Issue 17: Volume mount permissions (UID mismatch)
**Problem:** Files created by Airflow container (UID 50000) become unwritable on host. Stale files owned by host user cause `PermissionError` inside container.
**Root Cause:** Airflow image runs as `airflow` (UID 50000). Host files are owned by UID 1000. Bind mounts inherit host ownership.
**Fix:** Added `user: "50000:50000"` to both Airflow services in `docker-compose.yml`. Takes effect on next `docker compose restart`.
**Key Takeaway:** If you use bind mounts with a container that runs as a specific user, set `user:` in docker-compose to match, or use named volumes.
