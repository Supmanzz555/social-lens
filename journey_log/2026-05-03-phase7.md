# Phase 7: dbt Models — Implementation Log

## What was done
1.  **Configured dbt for Local Postgres:** Updated `dbt/profiles.yml` to point to `localhost:5432` and `analytics_db`.
2.  **Installed dbt-postgres:** `uv pip install dbt-postgres` in the project `.venv`.
3.  **Defined Sources:** Created `dbt/models/schema.yml` to register the Spark-generated tables (`gold_content_engagement`, etc.) as dbt sources.
4.  **Built Staging Models:** Created `stg_hackernews__stories`, `stg_github__events`, and `stg_youtube__videos` to clean and standardize the `gold_content_engagement` table per platform. Added missing columns (e.g., `avg_views`) to HN/GitHub models to ensure the `UNION ALL` in the intermediate model works.
5.  **Built Intermediate Model:** Created `int_daily_metrics` to union all staging models.
6.  **Built Mart Models:** Created `fct_daily_engagement`, `dim_creators`, and `dim_platforms`.
7.  **Built Tests:** Created `test_positive_scores` and `test_valid_dates`.
8.  **Verification:** 
    *   `dbt run`: All 7 models built successfully.
    *   `dbt test`: All 2 tests passed.
    *   `dbt docs generate`: Documentation created.
    *   Verified tables exist in Postgres (`fct_daily_engagement`, `dim_creators`, `dim_platforms`).

## Key Takeaway
dbt models act as the interface between "processed" data (from Spark) and "BI-ready" data (for Metabase). Defining `schema.yml` for pre-existing tables is crucial for dbt to recognize them as sources.
