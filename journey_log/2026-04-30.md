## 2026-04-30

### Issue 1: Docker Compose `version` attribute warning
- **Problem**: `docker-compose.yml` had `version: "3.8"` which Compose v2 deprecated
- **Fix**: Not critical, but will remove in cleanup
- **Status**: Ignored for now

### Issue 2: Airflow webserver — `ValueError: Unable to configure handler 'processor'`
- **Problem**: Airflow container exited immediately with logging configuration error. Root cause: the mounted `./airflow/logs` directory had wrong permissions; the `airflow` user inside the container couldn't create `/opt/airflow/logs/scheduler/` subdirectories
- **Fix**: `chmod -R 777 ./airflow/logs` on host
- **Key Takeaway**: Airflow's Docker image runs as user `airflow` (UID 50000). Any mounted volume for logs/dags/plugins must be writable by that UID. Always chmod or set user in docker-compose
- **Status**: Fixed

### Issue 3: Airflow scheduler — same logging error
- **Problem**: Scheduler service didn't have `AIRFLOW__LOGGING__REMOTE_LOGGING: "False"` env var, causing same handler configuration failure
- **Fix**: Added the same logging config to scheduler service in docker-compose.yml
- **Status**: Fixed

### Issue 4: Airflow webserver — Gunicorn timeout (120s)
- **Problem**: Gunicorn started but crashed with "No response from gunicorn master within 120 seconds". The Neon DB migration was slow (cloud DB + SSL), exceeding the default timeout
- **Fix**: Set `AIRFLOW__WEBSERVER__WEB_SERVER_MASTER_TIMEOUT: "300"` in docker-compose.yml
- **Status**: Fixed

### Issue 5: Airflow webserver — `connection reset by peer` / HTTP 000
- **Problem**: Webserver container appeared running but port returned no response. Container was actually in startup phase (DB migrate + user creation + permission seeding takes ~3-4 min on cold start)
- **Fix**: Just needed patience. After ~4 minutes, Gunicorn started listening
- **Key Takeaway**: Airflow cold start is slow, especially with remote cloud DB. Always check `docker logs` before assuming failure
- **Status**: Not a bug, just slow startup

### Issue 6: Kafka topics lost after `docker compose down`

- **Key Takeaway**: KRaft mode topic persistence may require proper volume mount of `/var/lib/kafka/data`. Need to investigate if the volume mount path is correct for this image version. Topics should be scripted in a startup init script

### Issue 9: Neon latency causing slow service initialization
- **Problem**: Airflow webserver took ~4 minutes to start. Metabase took ~14 minutes to finish initialization. Both use Neon PostgreSQL as their application database. Every DB operation (migration, permission seeding, table creation) is slowed by network latency to the cloud.
- **Root cause**: Neon is serverless PostgreSQL hosted in `us-east-2` (Ohio). User is in Thailand (Asia). Network RTT is ~200-300ms. Metabase runs 59+ Liquibase migrations on first start, each one a separate DB round-trip. Airflow does similar migration + permission seeding.
- **Impact**: 
  - Cold startup is very slow (not a runtime issue, only initialization)
  - Airflow health check: healthy after init, ~1s response time (acceptable)
  - Metabase health check: healthy after init, ~1ms response (acceptable)
  - Once initialized, services work fine
- **Workaround**: Patience. Services do work after initialization.
- **Future concern**: If Neon goes to sleep (serverless idle timeout), services may experience latency spikes on next query. This could affect Airflow DAG runs and Metabase dashboard loading.
- **Status**: Working but slow startup. Acceptable for Phase 1 but needs monitoring.

### Issue 10: Kafka topic alignment + auto-creation
- **Problem**: Plan used topic names `hackernews-posts`, `github-events`, `youtube-videos` but actual Kafka had `hn-raw`, `github-raw`, `yt-raw`. Also, topics disappeared on `docker compose down`.
- **Fix**: 
  1. Deleted old topics, created new ones matching the plan (3 partitions each).
  2. Added topic creation to Kafka container's `command`: starts server in background, waits for port 9092, creates topics with `--if-not-exists`, then `wait` to keep container alive.
  3. Used `kafka-storage.sh random-uuid > /tmp/cluster_id && kafka-storage.sh format -t $$(cat /tmp/cluster_id)` to avoid docker-compose `$VAR` interpolation issues.
  4. Updated plan's Phase 1 and Phase 2 verification steps to use `PLAINTEXT` instead of `SASL_SSL` (local Kafka has no auth).
- **Status**: Fixed — topics auto-created on every container start

### Issue 11: Moving Airflow + Metabase metadata DBs to local
- **Problem**: Neon PostgreSQL latency (200-300ms RTT to Ohio) caused slow initialization: Airflow took 4 min, Metabase took 14 min.
- **Fix**: Switched Airflow to SQLite (`sqlite:////opt/airflow/data/airflow.db`) with `SequentialExecutor`, Metabase to H2 (embedded, default).
- **Gotcha 1**: SQLite NOT compatible with `LocalExecutor` in Airflow 2.9.0. Downgraded to `SequentialExecutor` (no parallelism — fine for learning).
- **Gotcha 2**: Named volumes for SQLite had permission issues (airflow user UID 50000 can't write to root-owned Docker volumes). Named volumes also had directory creation issues. **Final fix**: bind mount to `./airflow/data/` on host with `chmod 777`.
- **Gotcha 3**: Docker-compose interpolates `$VAR` in command strings before passing to container. Use single quotes `'...'` around `bash -c` content, or `$$` escape.
- **Gotcha 4**: Airflow webserver command with `|| true` at the end caused silent failures — container appeared running but Airflow never started. Need to check logs, not just `docker ps`.
- **Result**: Airflow init: ~60 seconds. Metabase init: ~60 seconds. Zero extra RAM.
- **Status**: Fixed

### Issue 12: SQLite + SequentialExecutor not production-like enough
- **Problem**: `SequentialExecutor` means tasks run one at a time. No parallelism at all. This defeats the purpose of learning Airflow properly. Plan expects `LocalExecutor`.
- **Fix**: Added local PostgreSQL container (`postgres:16-alpine`, ~200MB RAM) for Airflow metadata. Switched back to `LocalExecutor`.
- **Gotcha 1**: Removed `airflow_db` Docker volume and `./airflow/data/` bind mount — no longer needed.
- **Gotcha 2**: Airflow services now `depends_on` Postgres with `condition: service_healthy`.
- **Result**: Full `LocalExecutor` with parallel task support. Total containers: 5. RAM: ~2-2.5GB.
- **Status**: Fixed

### Issue 17: Phase 2 — Producer implementation issues
- **Problem 1**: `configs/kafka_topics.json` had `replication_factor: 3` but we only have 1 broker.
- **Fix 1**: Changed to `replication_factor: 1`.

### Issue 18: Phase 3 — PySpark S3A FileSystem missing
- **Problem**: Spark streaming job failed with `ClassNotFoundException: org.apache.hadoop.fs.s3a.S3AFileSystem`. PySpark doesn't bundle Hadoop AWS JARs by default.
- **Fix**: Added `org.apache.hadoop:hadoop-aws:3.3.4` and `org.apache.hadoop:hadoop-common:3.3.4` to `spark.jars.packages` in `configs/spark_config.py`. These download automatically on first run (~300MB).
- **Gotcha 1**: Spark memory config was `2g` for driver and executor, which would exceed the 3GB system limit with 5 other containers running. Reduced to `512m` (batch) and `768m` (streaming).
- **Gotcha 2**: Checkpoints from failed runs cause issues on retry. Always clean `spark/streaming/checkpoints/` before re-running.
- **Gotcha 3**: `spark.sql.adaptive.enabled` is not supported in streaming DataFrames — Spark warns and disables it automatically.
- **Gotcha 4**: Java 21 works fine with Spark 3.5.1 despite official support listing Java 8/11/17. The `NativeCodeLoader` warning about native Hadoop library is harmless — Spark falls back to Java classes.
- **Result**: Streaming job reads from all 3 Kafka topics, writes partitioned Parquet to R2 `bronze/source=X/`. Schema verified: HackerNews 12 columns, data present. 8/8 Phase 3 checks passed.
