# Journey Log — 2026-05-02

## Phase 4 Issues

### Issue 1: System date mismatch (2569 vs 2026)
**Problem:** `date +%Y-%m-%d` returns `2569-05-02` because the system timezone is Asia/Bangkok which uses the Buddhist Era (BE) calendar, not Gregorian. The verify_phase4.sh script used this as the partition date, so bronze_to_silver.py looked for data in the wrong date.
**Root cause:** Thailand's default calendar year is Buddhist Era (2569 = 2026 + 543).
**Fix:** Changed bronze_to_silver.py, silver_to_gold.py, and gold_to_postgres.py to not require a partition date argument — they read all data from R2 regardless of date partitioning. Also fixed verify_phase4.sh to not pass `--date`.
**Key takeaway:** Always use Python's `datetime.date.today()` in code for dates, never rely on system `date` command.

### Issue 2: Bronze data not date-partitioned on disk
**Problem:** The batch scripts expected `bronze/source=X/partition_date=YYYY-MM-DD/` but Spark streaming wrote files directly to `bronze/source=X/` (no date subdirectory). The partition_date column exists in the Parquet schema but isn't used for filesystem partitioning.
**Root cause:** The streaming job uses `current_date()` as a column but writes to a flat path.
**Fix:** Changed `read_bronze()` to read `bronze/source={source}/` without date subdirectory. Same for silver and gold reads.
**Key takeaway:** Check actual R2 file structure before writing batch code that assumes a partition layout.

### Issue 3: Missing Hadoop S3A FileSystem in batch Spark sessions
**Problem:** `get_spark_session()` (used by batch jobs) didn't include `spark.jars.packages` for hadoop-aws/hadoop-common. Only `get_spark_with_kafka()` had them. Result: `ClassNotFoundException: org.apache.hadoop.fs.s3a.S3AFileSystem`
**Root cause:** The streaming config included S3A JARs, but the batch config only set `spark.hadoop.fs.s3a.*` properties without the actual JAR packages.
**Fix:** Added `spark.jars.packages` with `hadoop-aws:3.3.4` and `hadoop-common:3.3.4` to `get_spark_session()` in `configs/spark_config.py`.
**Key takeaway:** Both Spark session configs need the S3A JARs — properties alone aren't enough, the JARs must be on the classpath.

### Issue 4: Spark streaming metadata pointing to deleted files
**Problem:** Spark's `_spark_metadata/` directories in R2 contained references to Parquet files that were deleted in a previous cleanup. When Spark tried to read the bronze directory, it crashed with `FileNotFoundException`.
**Root cause:** Deleting individual Parquet files without clearing `_spark_metadata/` leaves stale references.
**Fix:** Deleted all `_spark_metadata/` directories from R2 bronze paths before running batch jobs.
**Key takeaway:** When cleaning up old Parquet files from R2, always delete the `_spark_metadata/` folders too, or use `df.write.mode("overwrite")` which handles it automatically.

### Issue 5: SparkSession chaining error in gold_to_postgres.py
**Problem:** `get_spark_session("gold-to-postgres").config("spark.jars.packages", ...)` fails because `SparkSession` has no `.config()` method — only `SparkSession.builder` does.
**Root cause:** Tried to chain `.config()` on a SparkSession object instead of the builder.
**Fix:** Rewrote gold_to_postgres.py to build the session directly with all config options in the builder chain.
**Key takeaway:** `.config()` is a builder method, not a SparkSession method. Use `SparkSession.builder.config().getOrCreate()`.

### Issue 6: Missing PostgreSQL JAR in gold_to_postgres.py
**Problem:** gold_to_postgres.py didn't include hadoop-aws packages, only `org.postgresql:postgresql:42.7.3`. Could read gold from R2 but failed with S3A ClassNotFoundException.
**Root cause:** Added Postgres JDBC but forgot S3A dependencies needed to read gold Parquet from R2 first.
**Fix:** Added `hadoop-aws:3.3.4,hadoop-common:3.3.4` to the `spark.jars.packages` list.
**Key takeaway:** gold_to_postgres.py needs both: S3A JARs (to read from R2) and PostgreSQL JDBC (to write to local Postgres).

### Issue 7: countDistinct import missing in silver_to_gold.py
**Problem:** `countDistinct` was used in build_creator_metrics but not imported from pyspark.sql.functions.
**Root cause:** Initial import line didn't include `countDistinct`.
**Fix:** Added `countDistinct` to the import list.
**Key takeaway:** Always verify all Spark functions are imported before using them.

### Issue 8: YouTube tag explode in silver_to_gold.py
**Problem:** silver_to_gold.py tried to `explode()` the `tag` column from YouTube silver data, but bronze_to_silver.py already exploded tags into a flat `tag` string column (not a list). Double explode failed.
**Root cause:** bronze_to_silver explodes tags during cleaning, so silver already has flat tag rows.
**Fix:** Removed the `explode()` in silver_to_gold.py and just filter/group on the existing `tag` column directly.
**Key takeaway:** Know what transformations each layer applies — silver already has exploded tags, gold doesn't need to explode again.

### Issue 9: MLflow dies on startup via start_mlflow.sh
**Problem:** `bash scripts/start_mlflow.sh` starts MLflow but it immediately receives SIGTERM and shuts down.
**Root cause:** The `nohup` wrapper in the script interacts poorly with the parent shell.
**Fix:** Run `uv run mlflow server --host 0.0.0.0 --port 5000 &>/tmp/mlflow.log &` directly instead of using the script.
**Key takeaway:** nohup + uv run can cause signal propagation issues. Run MLflow directly as a background process.

### Issue 10: PostgreSQL tables accumulate on re-run
**Problem:** gold_to_postgres.py uses `mode("append")` so running verification multiple times doubles/triples the row counts in PostgreSQL tables.
**Root cause:** Append mode doesn't deduplicate by date.
**Fix:** Not fixed yet — acceptable for development. For production, should add upsert logic or truncate before loading.
**Key takeaway:** Append mode is fine for Phase 4 testing but needs to be addressed before Airflow orchestration (Phase 5).

## Phase 5 Issues

### Issue 11: Airflow DAGs can't reach local Spark scripts
**Problem:** BashOperator in Airflow containers tried to run uv run python spark/batch/... but the container didn't have spark/, configs/, or .env mounted.
**Root cause:** The original plan had Spark running on the host, but Airflow runs in Docker. No bridge between them was defined.
**Fix:** Switched to SparkSubmitOperator with local mode. Added volume mounts for spark/, configs/, producers/, and .env to Airflow containers. Created custom Dockerfile.airflow to pre-install pyspark and pre-download JARs.
**Key takeaway:** The plan had a structural gap — Airflow (Docker) can't trigger host processes. Put everything Airflow needs inside the container.

### Issue 12: SparkSubmitOperator defaults to YARN when connection missing
**Problem:** SparkSubmitOperator(conn_id=spark_default) defaulted to --master yarn because the connection didn't exist.
**Root cause:** No Airflow connection named spark_default was created.
**Fix:** Created connection via airflow connections add spark_default --conn-type spark --conn-host local. Updated docker-compose.yml startup command.
**Key takeaway:** SparkSubmitOperator needs a valid connection or it defaults to yarn.

### Issue 13: Provider version conflicts (Airflow 3.2.1 pulled in)
**Problem:** Installing apache-airflow-providers-apache-spark without version pins pulled in Airflow 3.2.1, breaking Alembic migrations.
**Root cause:** pip dependency resolution upgraded Airflow.
**Fix:** Used --no-deps for providers, pinned versions (4.1.5, 1.8.0), built custom Dockerfile.
**Key takeaway:** Always pin provider versions. Use --no-deps when installing on top of fixed Airflow.

### Issue 14: Spark JAR downloads fail in container
**Problem:** spark-submit --packages timed out downloading JARs from Maven Central.
**Root cause:** Full transitive dependency tree takes too long to download.
**Fix:** Pre-downloaded 3 essential JARs (hadoop-aws, aws-java-sdk-bundle, postgresql) into Spark jars/ dir via Dockerfile. Removed spark.jars.packages from DAGs.
**Key takeaway:** Pre-build JARs into the image. Runtime --packages is unreliable.
